## 项目背景信息

FSAE， 输入是车上的CAN （可能需要接两路）和24V 电源和地。输出是给买的有人DTU （12V ,1A ,485模块，展锐 8910DM 方案），通讯是通过Protobuf （nanopb）+ MQTT（透传）到阿里云再通过grafana展示。

## DTU 相关配置

型号：USR-DR154（含蓝牙）
USR-DR154资料  ：https://www.usr.cn/wiki/puba/bKivFUA0Z
切换模式： “MQTT模式”

| **DTU 配置项**   | **值**            | **说明**                           |
| ------------- | ---------------- | -------------------------------- |
| **服务器地址/IP**  | `82.157.204.124` | 腾讯云公网 IP（bitfsae-com）             |
| **端口号**       | `1883`           | 标准 MQTT 非加密端口                    |
| **Client ID** | `123456`         | 必须唯一，建议改为 `FSAE_Vehicle_01` 之类的  |
| **用户名/密码**    | 保持为空             | 除非服务器开启了 `allow_anonymous false` |
| **发布主题 1**    | `fsae/telemetry` | 统一数据通道，基础遥测和 BMS 详细数据都走这个 Topic |
| **QoS 等级**    | `0` 或 `1`        | 赛车数据实时性高，选 `0`（尽力交付，延迟最低）        |
MQTT 协议版本：选 `3.1.1`（比 3.1 更稳定，且主流 Broker 都支持）。
开启清除会话（Clean Session）：没必要重传历史旧数据。
无数据&无连接重启时间：缩短到 `300s`。如果车在场上信号断了，5 分钟不自动重连成功就重启 DTU，有助于恢复网络。

补充：如果终端设备只能配置一个发布主题，就直接使用 `fsae/telemetry`；BMS 详细数据通过同一条 Protobuf `TelemetryFrame` 中的 `modules` 字段上送。

当前本地联调也支持用 `local_sim2.py --mode serial` 通过 USB 转 485 直接给 DTU 喂同一份 Protobuf 数据，便于在车端板子未完成时先验证 DTU 链路。
