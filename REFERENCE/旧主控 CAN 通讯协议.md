## 1. 总线配置

### 1.1 CAN1

**用途**：旧主控与从控板、显示端、参数配置工具通信

| 项目 | 值 | 说明 |
|---|---|---|
| 引脚 | PA11(RX), PA12(TX) | STM32 CAN1 |
| 波特率 | 250 kbps | 固定 |
| 帧类型 | 扩展帧 | 代码只发送扩展帧；接收逻辑也只按扩展 ID 分发 |
| 接收 FIFO | FIFO0 | |
| 中断 | `CAN1_RX0_IRQn` | 抢占优先级 0，响应优先级 1 |
| 过滤器 | 全通 | Mask 全 0，接收所有 ID |
| 自动离线恢复 | 关闭 | `ABOM=DISABLE` |
| 自动重传 | 禁用 | `NART=ENABLE` |

**初始化参数**

```c
CAN1_Mode_Init(CAN_SJW_1tq, CAN_BS2_2tq, CAN_BS1_3tq, 24, CAN_Mode_Normal);
// 36MHz / ((1 + 3 + 2) * 24) = 250kbps
```

### 1.2 CAN2

**用途**：旧主控与充电机或整车 ECU 通信

| 模式 | 条件 | 波特率 | 帧类型 | 接收 FIFO | 说明 |
|---|---|---:|---|---|---|
| 充电模式 | `CurrentChargeMode != 0` | 250 kbps | 扩展帧收发 | FIFO1 | 与充电机通信 |
| 放电模式 | `CurrentChargeMode == 0` | 500 kbps | 标准帧发送 | FIFO1 | 仅发送给 ECU |

**引脚**：PB5(RX), PB6(TX)，CAN2 重映射  
**中断**：`CAN2_RX1_IRQn`

**初始化参数**

```c
// 充电模式
CAN2_ChargeMode_Init(CAN_SJW_1tq, CAN_BS2_3tq, CAN_BS1_2tq, 24, CAN_Mode_Normal);
// 36MHz / ((1 + 2 + 3) * 24) = 250kbps

// 放电模式
CAN2_DischargeMode_Init(CAN_SJW_1tq, CAN_BS2_2tq, CAN_BS1_3tq, 12, CAN_Mode_Normal);
// 36MHz / ((1 + 3 + 2) * 12) = 500kbps
```

**模式选择规则**

- 上电时根据 `ChargingGunSignal` 只判定一次。
- `ChargingGunSignal == 0`：充电模式。
- `ChargingGunSignal != 0`：放电模式。
- 运行中不会因为 CAN 命令切换 CAN2 工作模式。

## 3. 发送时序

### 3.1 调度方式

- 旧主控的 CAN 周期发送在 `TIM4_IRQHandler()` 中断里直接执行。
- `TIM4` 周期为 50 ms。
- 中断里存在多次 `delay_ms(1)`，因此报文间固定插入约 1 ms 间隔。
- 这与新主控“中断只记时，主循环执行发送”不同。

### 3.2 CAN1 周期发送

**`CANCounter % 10 == 0`，即每 500 ms 发送以下 7 帧，顺序固定：**

1. `0x186050F4`
2. `0x186150F4`
3. `0x186250F4`
4. `0x186350F4`
5. `0x187650F4`
6. `0x187750F4`
7. `0x187F50F4`

每帧之间插入 `delay_ms(1)`。

### 3.3 CAN1 均衡状态发送

- 条件：`CellBalSwitch == 1`
- 节拍：`CANCounter % 5 == 0`
- 实际周期：250 ms
- 每次连续发送 3 帧：
  - `0x186450F4`
  - `0x186550F4`
  - `0x186650F4`

### 3.4 CAN2 发送

**放电模式**

- `CANCounter % 10 == 0`，每 500 ms 发送 `0x401`

**充电模式**

- `CANCounter % 20 == 0`，每 1 s 发送 `0x1806E5F4`
- 上高压开始时立即发送一次“允许充电”
- 下高压时立即发送一次“停止充电”

## 4. 字节序与单位约定

### 4.1 字节序

| 报文类别 | 字节序 | 说明 |
|---|---|---|
| CAN1 主控发送多字节整数 | 大端 | 高字节在前 |
| CAN1 从控电压 | 小端 | 低字节在前 |
| CAN1 从控温度 | 单字节 | 无大小端问题 |
| CAN2 充电请求/反馈 | 大端 | `0x1806E5F4`、`0x18FF50E5` |
| CAN2 放电状态 `0x401` | 小端 | Intel 格式 |

### 4.2 单位与编码

| 物理量 | 变量 | 单位 | 编码方式 | 说明 |
|---|---|---|---|---|
| 总压 | `BatVoltage` | 0.1 V | 直接值 | `5700` 表示 570.0V |
| 预充电压 | `PreBatVoltage` | 0.1 V | 直接值 | |
| 单体电压 | `CellVolt[]` | mV | 直接值 | |
| 最大/最小单体电压序号 | `MaxCellVoltNo/MinCellVoltNo` | 序号 | 0 基 | 显示端若要显示“第几个”，需 `+1` |
| 温度 | `CellTemp[]` | °C | 偏移 +30 | `55` 表示 25°C |
| 温度序号 | `MaxTempNo/MinTempNo` | 序号 | 0 基 | |
| 电流 | `BatCurrent` | 0.1 A | 偏移 10000 | `10000` 为 0A |
| 充电机反馈电流 | `ChrBatCurrent` | 0.1 A | 直接值 | 只在充电模式总览帧使用 |
| 功率 | `BatPower` | W | 偏移 1000 | `<1000` 表示充电，`>1000` 表示放电 |
| SOC | `BatSoc` | % | 直接值 | |
| 充电请求电压 | `ChrReqVolt` | 0.1 V | 直接值 | |
| 充电请求电流 | `ChrReqCurr` | 0.1 A | 直接值 | |

### 4.3 电流与功率公式

```c
// 电流原始值 -> 物理值
current_A = (BatCurrent - 10000) / 10.0;

// 功率原始值 -> 物理值
if (BatPower > 1000) {
    power_W = BatPower - 1000;   // 放电
} else {
    power_W = BatPower - 1000;   // 充电时为负数
}
```

**注意**

- 旧主控的电流来自本板 ADC，不是 CAN 传感器。
- 调试串口里出现过 `((BatCurrent-10000)*2.4)` 的显示换算，这只是串口打印，不影响 CAN 发出的实际编码。

## 5. CAN1 接收详解

### 5.1 从控电压帧

#### 5.1.1 ID 规则

电压基址：

```text
0x180050F3
```

旧主控共接收 6 个从控，每个从控占 6 个扩展 ID。

| 从控序号 | ID 范围 |
|---|---|
| 从控 1 | `0x180050F3` ~ `0x180550F3` |
| 从控 2 | `0x180650F3` ~ `0x180B50F3` |
| 从控 3 | `0x180C50F3` ~ `0x181150F3` |
| 从控 4 | `0x181250F3` ~ `0x181750F3` |
| 从控 5 | `0x181850F3` ~ `0x181D50F3` |
| 从控 6 | `0x181E50F3` ~ `0x182350F3` |

#### 5.1.2 单个从控的 6 帧映射

每个从控管理 23 节单体。6 帧的解析规则不是整齐的 4 节/帧，而是：

- 第 1 帧：只使用 Byte2~Byte7，携带 3 节单体
- 第 2~6 帧：使用 Byte0~Byte7，每帧 4 节单体

因此每个从控总计：

```text
3 + 4 + 4 + 4 + 4 + 4 = 23 节
```

#### 5.1.3 字段格式

所有单体电压均为 **小端 16 位**，单位 mV。

**第 1 帧**

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | 旧主控忽略，不参与解析 |
| Byte2~Byte3 | 第 1 节单体电压 |
| Byte4~Byte5 | 第 2 节单体电压 |
| Byte6~Byte7 | 第 3 节单体电压 |

**第 2~6 帧**

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | 1 节单体电压 |
| Byte2~Byte3 | 1 节单体电压 |
| Byte4~Byte5 | 1 节单体电压 |
| Byte6~Byte7 | 1 节单体电压 |

#### 5.1.4 每个从控的单体编号

| 从控 | 帧 0 | 帧 1 | 帧 2 | 帧 3 | 帧 4 | 帧 5 |
|---|---|---|---|---|---|---|
| 从控 1 | 1~3 | 4~7 | 8~11 | 12~15 | 16~19 | 20~23 |
| 从控 2 | 24~26 | 27~30 | 31~34 | 35~38 | 39~42 | 43~46 |
| 从控 3 | 47~49 | 50~53 | 54~57 | 58~61 | 62~65 | 66~69 |
| 从控 4 | 70~72 | 73~76 | 77~80 | 81~84 | 85~88 | 89~92 |
| 从控 5 | 93~95 | 96~99 | 100~103 | 104~107 | 108~111 | 112~115 |
| 从控 6 | 116~118 | 119~122 | 123~126 | 127~130 | 131~134 | 135~138 |

#### 5.1.5 在线判定

- 旧主控收到某从控任意一帧**电压帧**时，会把该从控在线计数器清零。
- 温度帧不会刷新在线计数器。
- 因此旧主控对“从控在线”的判断，本质上只看电压帧是否持续到达。

### 5.2 从控温度帧

#### 5.2.1 ID 规则

温度基址：

```text
0x184050F3
```

| 从控序号 | ID |
|---|---|
| 从控 1 | `0x184050F3` |
| 从控 2 | `0x184150F3` |
| 从控 3 | `0x184250F3` |
| 从控 4 | `0x184350F3` |
| 从控 5 | `0x184450F3` |
| 从控 6 | `0x184550F3` |

#### 5.2.2 数据格式

温度为单字节，编码方式为：

```text
实际温度 = 字节值 - 30
```

#### 5.2.3 各从控温度映射

| 从控 | 解析规则 | 有效温度数 |
|---|---|---:|
| 从控 1 | 使用 Byte0~Byte7 | 8 |
| 从控 2 | 使用 Byte0~Byte7 | 8 |
| 从控 3 | 使用 Byte1~Byte7，Byte0 忽略 | 7 |
| 从控 4 | 使用 Byte1~Byte7，Byte0 忽略 | 7 |
| 从控 5 | 使用 Byte0~Byte7 | 8 |
| 从控 6 | 使用 Byte0~Byte7 | 8 |

#### 5.2.4 编号范围

| 从控 | 温度编号 |
|---|---|
| 从控 1 | 1~8 |
| 从控 2 | 9~16 |
| 从控 3 | 17~23 |
| 从控 4 | 25~31 |
| 从控 5 | 33~40 |
| 从控 6 | 41~48 |

**注意**

- 旧代码数组是 48 长度，但从控 3、4 各只更新 7 个位置。
- 对应数组中的第 24、32 个位置不会被这些温度帧刷新。
- 这是旧主控代码的实际行为，接收端不要假设 6 个从控都是完整 8 路温度。

### 5.3 工具参数写入帧 `0x188050F5`

**功能**：修改充电请求电压、电流

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `ChrReqVolt`，大端，0.1V |
| Byte2~Byte3 | `ChrReqCurr`，大端，0.1A |
| Byte4~Byte7 | 旧主控未使用 |

**实际行为**

- 收到后立刻更新 RAM 中的 `ChrReqVolt`、`ChrReqCurr`
- 置位 `WriteRomFlag`
- 主循环稍后把新值写入外部 Flash
- **没有 CAN 应答帧**

### 5.4 工具告警阈值写入帧 `0x188150F5`

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | 单体过压阈值 `ALM_CELL_OV_VALUE`，mV，大端 |
| Byte2~Byte3 | 单体欠压阈值 `ALM_CELL_UV_VALUE`，mV，大端 |
| Byte4 | 单体高温阈值 `ALM_CELL_OT_VALUE`，编码值，实际温度=`值-30` |
| Byte5 | 单体低温阈值 `ALM_CELL_UT_VALUE`，编码值，实际温度=`值-30` |
| Byte6~Byte7 | 未使用 |

**实际行为**

- 收到后立即更新 RAM
- 置位 `WriteRomFlag`
- 主循环稍后写 Flash
- **没有 CAN 应答帧**

### 5.5 工具告警开关写入帧 `0x188250F5`

**Byte0**

| 位 | 含义 |
|---|---|
| bit7 | 单体过压开关 |
| bit6 | 单体欠压开关 |
| bit5 | 单体高温开关 |
| bit4 | 单体低温开关 |
| bit3 | 电池压差开关 |
| bit2 | 电池温差开关 |
| bit1 | 稳态充电过流开关 |
| bit0 | 稳态放电过流开关 |

**Byte1**

| 位 | 含义 |
|---|---|
| bit7 | 从控离线开关 |
| bit6 | 预充失败开关 |
| bit5 | 高压异常开关 |
| bit4 | 霍尔断线开关 |
| bit3~bit0 | 未使用 |

**实际行为**

- 收到后立即更新 RAM
- 置位 `WriteRomFlag`
- 主循环稍后写 Flash
- **没有 CAN 应答帧**

## 6. CAN1 发送详解

### 6.1 总览帧 `0x186050F4`

DLC=7，大端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `BatVoltage`，总压，0.1V |
| Byte2~Byte3 | 电流 |
| Byte4 | `BatSoc` |
| Byte5 | `IMD_signal` |
| Byte6 | 高 4 位=`BatState`，低 4 位=`BatAlmLv` |

**Byte2~Byte3 的来源**

- 充电模式：发送 `ChrBatCurrent`
- 放电模式：发送 `BatCurrent`

**`BatState` 取值**

| 值 | 含义 |
|---:|---|
| 2 | 自检 |
| 3 | 待机 |
| 4 | 预充 |
| 5 | 已上高压，具体是充电还是放电需结合 `CurrentChargeMode` |
| 7 | 故障 |

**`BatAlmLv` 取值**

- `0`：无报警
- `1`：存在一级报警
- `2`：存在二级报警

### 6.2 电压极值帧 `0x186150F4`

DLC=6，大端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `MaxCellVolt`，mV |
| Byte2~Byte3 | `MinCellVolt`，mV |
| Byte4 | `MaxCellVoltNo`，0 基 |
| Byte5 | `MinCellVoltNo`，0 基 |

### 6.3 温度极值帧 `0x186250F4`

DLC=5

| 字节 | 含义 |
|---|---|
| Byte0 | `MaxTemp`，编码值，实际温度=`值-30` |
| Byte1 | `MinTemp`，编码值，实际温度=`值-30` |
| Byte2 | `MaxTempNo`，0 基 |
| Byte3 | `MinTempNo`，0 基 |
| Byte4 | `CoolingCtl`，风扇控制状态 |

### 6.4 继电器/充电状态帧 `0x186350F4`

DLC=8，大端

**Byte0**

| 位段 | 含义 |
|---|---|
| bit7~6 | `PosRlyStr` |
| bit5~4 | `NegRlyStr` |
| bit3~2 | `PreRlyStr` |
| bit1~0 | 固定 0 |

实际使用中上述继电器状态变量只取 `0/1`。

**Byte1**

| 位段 | 含义 |
|---|---|
| bit7~4 | `ChrState` |
| bit3 | `ChrCommunication` |
| bit2~0 | 固定 0 |

**Byte2~Byte7**

| 字节 | 含义 |
|---|---|
| Byte2~Byte3 | `ChrReqVolt`，0.1V |
| Byte4~Byte5 | `ChrReqCurr`，0.1A |
| Byte6~Byte7 | `PreBatVoltage`，0.1V |

**`ChrState` 在代码中的实际取值**

| 值 | 含义 |
|---:|---|
| 0 | 放电模式 |
| 1 | 上电后进入充电模式，尚未进入周期充电控制 |
| 3 | 充电中 |
| 4 | 代码注释中表示充满，但当前主路径未实际走到 |

**`ChrCommunication`**

- 代码中定义了该变量并发送到 Byte1 bit3
- 旧代码未找到稳定更新逻辑
- 实际对接时应按保留位处理

### 6.5 告警状态帧 `0x187650F4`

DLC=6

每个告警字段占 2 bit，旧代码实际使用值：

- `0`：无告警
- `1`：一级告警
- `2`：二级告警

**Byte0**

| 位段 | 含义 |
|---|---|
| bit7~6 | 单体过压 |
| bit5~4 | 单体欠压 |
| bit3~2 | 单体高温 |
| bit1~0 | 单体低温 |

**Byte1**

| 位段 | 含义 |
|---|---|
| bit7 | 单体电压断线 |
| bit6 | 单体温度断线 |
| bit5~4 | 电池压差过大 |
| bit3~2 | 电池温差过大 |
| bit1~0 | 电池总压过高 |

**Byte2**

| 位段 | 含义 |
|---|---|
| bit7~6 | 电池总压过低 |
| bit5~4 | SOC 过高 |
| bit3~2 | SOC 过低 |
| bit1~0 | 稳态充电过流 |

**Byte3**

| 位段 | 含义 |
|---|---|
| bit7~6 | 稳态放电过流 |
| bit5~4 | 瞬态充电过流 |
| bit3~2 | 瞬态放电过流 |
| bit1~0 | 固定 0 |

**Byte4**

| 位段 | 含义 |
|---|---|
| bit7~5 | 固定 0 |
| bit4 | 预充失败 |
| bit3 | 继电器故障 |
| bit2 | 从控 5 离线标志 |
| bit1 | 从控 6 离线标志 |
| bit0 | 固定 0 |

**Byte5**

| 位段 | 含义 |
|---|---|
| bit7 | 高压异常 |
| bit6 | 霍尔断线 |
| bit5 | 从控整体离线总标志 |
| bit4 | 从控 1 离线标志 |
| bit3 | 从控 2 离线标志 |
| bit2 | 从控 3 离线标志 |
| bit1 | 从控 4 离线标志 |
| bit0 | 固定 0 |

### 6.6 告警阈值帧 `0x187750F4`

DLC=6，大端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | 单体过压阈值，mV |
| Byte2~Byte3 | 单体欠压阈值，mV |
| Byte4 | 单体高温阈值，编码值 |
| Byte5 | 单体低温阈值，编码值 |

### 6.7 告警开关帧 `0x187F50F4`

DLC=2

定义与接收命令 `0x188250F5` 完全一致：

- Byte0：8 个一级告警开关
- Byte1：4 个附加告警开关

### 6.8 均衡状态帧 `0x186450F4` ~ `0x186650F4`

每帧 DLC=6，共 3 帧。

| ID | 数据来源 |
|---|---|
| `0x186450F4` | `CellBalanceN[0..5]` |
| `0x186550F4` | `CellBalanceN[6..11]` |
| `0x186650F4` | `CellBalanceN[12..17]` |

每个字节是 8 位位图。旧代码按从控分区组织：

- 每个从控占 3 字节
- 每 bit 对应一个单体
- bit=1 表示该单体当前被置入均衡位图

## 7. CAN2 详解

### 7.1 充电请求帧 `0x1806E5F4`

DLC=5，大端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `ChrReqVolt`，0.1V |
| Byte2~Byte3 | `ChrReqCurr`，0.1A |
| Byte4 | `ChrCtl` |

**`ChrCtl` 实际含义**

| 值 | 含义 |
|---:|---|
| 0 | 允许充电 |
| 1 | 停止充电 |

**特殊行为**

- 周期发送时，如果 `Can2_STOP_Sent_BatCharge == 1`，则强制发停止充电：
  - Byte0~Byte3 清零
  - Byte4=1
- `HV_ACC_UP()` 会立刻发一次允许充电
- `HV_ACC_DOWM()` 会立刻发一次停止充电

### 7.2 充电机反馈帧 `0x18FF50E5`

DLC=5，大端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `ChrBatVoltage`，0.1V |
| Byte2~Byte3 | `ChrBatCurrent`，0.1A |
| Byte4 | `ChrBatState` |

`ChrBatState` 被旧主控原样保存。代码里曾尝试按位解释，但实现不完整；对接时应以整字节原样上报为准。

### 7.3 放电状态帧 `0x401`

DLC=8，小端

| 字节 | 含义 |
|---|---|
| Byte0~Byte1 | `BatVoltage`，0.1V |
| Byte2~Byte3 | `BatPower`，W，偏移 1000 |
| Byte4~Byte5 | `BatCurrent`，0.1A，偏移 10000 |
| Byte6 | `BatSoc` |
| Byte7 | `MaxTemp`，编码值，实际温度=`值-30` |

**说明**

- 旧主控没有 `0x402` 故障诊断帧。
- 放电模式下 CAN2 只主动发送 `0x401`。

### 7.4 放电模式接收

- 旧代码为 CAN2 放电模式配置了精确匹配过滤器。
- 但放电模式下未实现有效的 CAN2 接收分发逻辑。
- 实际对接可以按“放电模式只发不收”理解。

## 8. 收发职责与状态关联

### 8.1 旧主控接收谁的数据

| 总线 | 数据来源 | 内容 |
|---|---|---|
| CAN1 | 6 个从控 | 单体电压、温度 |
| CAN1 | 配置工具 | 充电参数、告警阈值、告警开关 |
| CAN2 | 充电机 | 充电反馈 |

### 8.2 旧主控发送什么

| 总线 | 发送对象 | 内容 |
|---|---|---|
| CAN1 | 显示端/上位机 | 总览、极值、继电器状态、告警、均衡 |
| CAN2 | 充电机 | 充电请求 |
| CAN2 | ECU | 放电状态 |

### 8.3 参数修改生效时机

- `0x188050F5`、`0x188150F5`、`0x188250F5` 收到后，RAM 立即更新。
- Flash 持久化不在 CAN 中断中做，而是在主循环里看到 `WriteRomFlag==1` 后执行。
- 主循环有 `delay_ms(1000)` 调试打印，因此持久化通常会有最多约 1 s 延迟。
- 旧主控**不会回发确认帧**。

### 8.4 报警与状态帧关系

- `0x187650F4` 反映各个 2 bit 告警值。
- `0x186050F4` 的 `BatAlmLv` 只给出整包当前最高报警等级。
- `0x186350F4` 给出高压继电器和充电状态。

## 9. 与新主控的差异

### 9.1 CAN1 接收侧差异

| 项目 | 旧主控 | 新主控 |
|---|---|---|
| 电流来源 | 本板 ADC，本地采样 | CAN1 标准帧 `0x03C0` |
| 电压从控帧 | 6 从控，共 138 节，首帧只用 Byte2~7 | 新协议不同，按新文档执行 |
| 温度从控帧 | 从控 3、4 仅 7 字节有效 | 新协议已重新定义 |
| 参数工具帧 | 只有 `0x188050F5`、`0x188150F5`、`0x188250F5` | 新主控新增校准、RTC、故障复位、电流方向等命令 |
| 长度校验 | 无显式 DLC 最小值校验 | 新主控有最小长度检查 |

### 9.2 CAN1 发送侧差异

| 项目 | 旧主控 | 新主控 |
|---|---|---|
| 周期发送位置 | `TIM4_IRQHandler()` 中断内直接发送 | 中断只记时，主循环发送 |
| 中断内延时 | 有，多次 `delay_ms(1)` | 无这种做法 |
| 总览帧数 | 7 个基础状态帧 + 3 个均衡帧 | 新主控更多，包括单体累加值、IMD 诊断等 |
| 应答帧 | 无 | 新主控有 RTC/校准/方向设置应答 |
| `0x186750F4` | 无 | 有 |
| `0x186850F4` | 无 | 有 IMD 诊断 |

### 9.3 CAN2 差异

| 项目 | 旧主控 | 新主控 |
|---|---|---|
| 放电模式 | 只发 `0x401` | 新主控发 `0x401` 和 `0x402` |
| 充电请求 | `0x1806E5F4`，1 s 周期，并在上/下高压时立即发 | 新主控实现方式不同 |
| 充电反馈 | `0x18FF50E5` | 新主控同样有充电总线，但实现已重构 |
| 自动恢复 | `ABOM=DISABLE` | 新主控 CAN1 开启自动离线恢复 |

### 9.4 模式切换差异

- 旧主控的充/放电模式在上电时依据充电枪信号决定。
- 旧主控收到 CAN 参数帧不会切换 CAN2 波特率和模式。
- 新主控的模式、发送任务、服务执行方式均已重构，不能把旧主控的运行节奏直接套到新主控。

## 10. 对接注意事项

1. 旧主控 CAN1 的从控电压首帧不是 4 节，而是 3 节，且在 Byte2~Byte7；Byte0~Byte1 会被旧主控忽略。
2. 旧主控温度帧中，从控 3、4 的 Byte0 会被忽略，只取 Byte1~Byte7。
3. 旧主控工具写入命令没有确认回帧，工具端需要自己超时管理。
4. 旧主控参数写入是“先改 RAM，后异步写 Flash”，掉电时机不合适可能导致 Flash 尚未落盘。
5. 旧主控放电模式 CAN2 只有 `0x401`，不能按新主控的 `0x401 + 0x402` 设计接收。
6. 旧主控的 CAN 周期发送发生在中断里，且带阻塞延时，实际时间抖动会比新主控更大。
7. 旧主控没有 `0x03C0` 电流传感器 CAN 帧，也没有新主控那组 `0x18A*` 校准/RTC/方向设置报文。
